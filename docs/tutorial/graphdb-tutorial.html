Obsidian-KDB Architecture Tutorial

GraphDB-KDB: the queryable memory behind the compiled knowledge base

GraphDB-KDB is the Kuzu-backed ontology layer for Obsidian-KDB. The Markdown wiki is the readable surface; the graph is the live structure that stores entities, sources, links, provenance, domains, aliases, and eventually belief-revision claims.

1. Mental Model

The project is best understood as a raw-text to knowledge-graph compiler. The compiler reads sources from KDB/raw/, asks the model to emit structured page intents, validates and canonicalizes them, writes Markdown for Obsidian, then syncs the same structured result into GraphDB-KDB.

Longer term, there can be multiple source-preparation pipelines upstream of this compiler. Most of those pipelines will produce enriched raw text and place it into KDB/raw/. The Obsidian vault is a special case: because the vault itself hosts KDB/raw/, vault-native sources can stay in place and be enriched in place, while KDB/raw/ remains the compiler's source boundary.

Core distinction: Obsidian is the IDE and human-readable rendering. GraphDB-KDB is the queryable memory of relationships.

3. Components

Schema

schema.py defines Kuzu node and relationship tables plus migrations.

Connection manager

graphdb.py opens Kuzu, bootstraps schema, and exposes the public API.

Graph write engine

ingestor.py is named from the graph's point of view: it applies one compiler result into Kuzu. This is different from the upstream ingestion pipelines that prepare enriched raw text before the compiler sees it.

Adapter

adapters/obsidian_runs.py translates Obsidian-KDB's run files into graph writes. A run file says "these pages and links were produced"; the adapter turns that into the specific Entity, LINKS_TO, and SUPPORTS operations GraphDB-KDB understands.

Queries

queries.py answers graph questions: what is near this entity, what links into it, what route connects two entities, which sources support it, and what custom Cypher query should Kuzu run.

Analytics

analytics.py computes graph-level signals: central entities with PageRank, clusters with Louvain community detection, and weak bridges with structural-hole detection.

Verifier and rebuilder

verifier.py and rebuilder.py are the recovery proof. The intended recovery path is replaying saved run sidecars into a clean Kuzu graph, then checking that replayed graph against the live graph.

CLI

cli.py powers the graphdb-kdb command-line interface.

4. Graph Basics

A Kuzu GraphDB is easiest to picture as a property graph with three pieces: nodes, edges, and properties. Nodes are the circles, squares, triangles, or other objects. Edges are the labeled lines between them. Properties are the data attached to either nodes or edges.

Source node: one raw source file Entity node: one summary, concept, article, or alias page Domain node: classification bucket Claim node: belief/assertion

Node

A node is one stored object in the graph. Node types are the graph equivalent of classes in the schema: Entity, Source, Domain, and Claim. Each type has its own allowed properties.

Connection

A connection is always between two nodes. The two nodes can be the same type or different types, but only if the schema allows that edge type. For example, rationality is an Entity node whose page_type is concept. If it links to mungers-four-levels-of-rationality, another Entity node, the graph-write code creates a LINKS_TO edge between them. If a raw source supports rationality, that is a SUPPORTS edge from a Source node to an Entity node.

Edge

An edge sits between two nodes. It does not live "on" either node. It is a separate relationship record with a start node, an end node, a type, and optional properties like run_id or created_at. The Markdown wikilink [[mungers-four-levels-of-rationality]] becomes a LINKS_TO edge from the page containing the wikilink to the page being linked.

Properties

Properties are data fields attached to nodes or edges. A node property might be Entity.slug = "rationality" or Entity.page_type = "concept". An edge property might be LINKS_TO.run_id, which records which compile run created that relationship.

Other Terms

  • slug: stable ID for an entity.
  • canonical_id: alias/canonicalization pointer; null means self-canonical.
  • neighbor: query result for a node reached by following one or more edges.
  • depth: number of hops to traverse.
  • subgraph: selected nodes plus the selected edges among those nodes.
  • provenance: standard data/knowledge-graph term for origin or evidence trail. In this tutorial, it means source-to-entity support metadata: "which source produced or supports this entity?"
  • orphan: an entity with no current source support.
  • community: an algorithmically detected cluster with denser internal links than external links.
  • structural hole: a weak bridge between communities.

5. Current Schema

Obsidian's graph view can feel like one kind of node connected by one kind of line. Kuzu is a typed graph: nodes have different types, and only schema-declared edge types can connect them. The visual picture is still a mesh, but the mesh has labeled node classes and labeled relationship classes.

Yes: each generated summary, concept, article, and alias page is one Entity node. Each raw source is one Source node. A source connects to every generated page it supports through SUPPORTS edges. The generated pages then connect to each other through LINKS_TO edges, which are the graph form of wikilinks. 

Source --SUPPORTS--> Entity --LINKS_TO--> Entity
Entity --BELONGS_TO--> Domain
Entity --ALIAS_OF--> Entity
Source --EVIDENCES--> Claim --ABOUT--> Entity
Claim --SUPERSEDES / CONTRADICTS / QUALIFIES--> Claim

The important shift: an edge is not a property of a node. It is its own typed record. That is what lets queries ask relationship questions directly, such as "which sources support this entity?" or "what entities point into this entity?"

Type Name Meaning
Node Entity Compiled knowledge object: summary, concept, article, or alias.
Node Source Raw input source with hash, path, status, source type, summary, author, and domain fields.
Node Domain Normalized classification bucket.
Node Claim Versioned belief/assertion used by the promotion and belief-revision layer.
Edge LINKS_TO Entity-to-entity wikilink relationship.
Edge SUPPORTS Source-to-entity provenance relationship.
Edge ALIAS_OF Alias entity to canonical entity.
Edge BELONGS_TO Entity-to-domain classification.
Edge EVIDENCES, ABOUT, SUPERSEDES, CONTRADICTS, QUALIFIES Claim-layer relationships for belief revision.

6. How GraphDB Is Populated

  1. kdb-scan identifies changed raw sources.
  2. The compiler produces structured JSON page intents.
  3. Validation rejects malformed compile output before writes.
  4. Reconciliation and canonicalization normalize slugs and links.
  5. Python writes Markdown pages and state journals.
  6. Stage 10 archives sidecars and syncs the canonical compile result into GraphDB.
Plain version of the adapter boundary: the compiler produces run data; the adapter translates that run data; the GraphDB core writes Kuzu records. The compiler does not open Kuzu and hand-write tables itself because that would mix producer-specific logic with graph-storage logic. 
compile_result pages/outgoing_links
  -> ObsidianRunsAdapter translates producer-shaped JSON
  -> graph write engine creates Entity, LINKS_TO, SUPPORTS records

7. Capabilities

  • Lookup central concepts via PageRank, which ranks entities by graph importance based on incoming and outgoing link structure.
  • Traverse nearby concepts with directed or bidirectional neighbors. A neighbor is reached by following edges from a starting node.
  • Find incoming references to a concept. An incoming reference is another entity with a LINKS_TO edge pointing into the current entity.
  • Find shortest paths between entities. A shortest path is the minimum number of edge hops between two nodes, subject to direction and max-hop settings.
  • Extract all entities created from a source through SUPPORTS provenance edges.
  • Inspect a source-specific subgraph: the selected source's supported nodes plus links among those nodes.
  • Detect orphan candidates.
  • Detect topic communities with Louvain clustering: groups of entities that link more densely to each other than to the rest of the graph.
  • Detect structural holes between communities: sparse bridges where a missing concept or source might improve the knowledge structure.
  • Rebuild from run history.
  • Verify live graph against replayed graph.
  • Snapshot graph state into diffable JSONL backup.

8. Build And Populate

Install the project in editable mode:

cd ~/Droidoes/Obsidian-KDB
pip install -e ".[dev]"

Initialize the graph schema:

graphdb-kdb init

Populate through the normal compile pipeline:

kdb-compile

Rebuild the graph from historical sidecars:

graphdb-kdb rebuild --vault-root ~/Obsidian --yes

Verify the live graph:

graphdb-kdb verify --vault-root ~/Obsidian

Snapshot the graph:

graphdb-kdb snapshot --vault-root ~/Obsidian
Operational note: avoid running multiple graphdb-kdb commands in parallel against the same Kuzu file. Kuzu can reject concurrent opens with a file-lock error.

9. Query Examples

Basic counts:

graphdb-kdb stats

Bidirectional depth-1 neighborhood around rationality:

graphdb-kdb neighbors rationality --direction both --depth 1

Incoming links to corruption-in-governance:

graphdb-kdb incoming corruption-in-governance

Shortest directed path:

graphdb-kdb path rationality value-investing --max-hops 5

Top PageRank entities:

graphdb-kdb pagerank --top 10

Community assignments:

graphdb-kdb communities

Structural holes:

graphdb-kdb structural-holes

Source-specific subgraph:

graphdb-kdb subgraph-by-source "KDB/raw/Li_Lu_Munger_College_Dec_2024.md"

Ad-hoc Cypher:

graphdb-kdb cypher "MATCH (e:Entity) RETURN e.slug, e.page_type ORDER BY e.slug LIMIT 10"

10. Practical Use Cases

Compile context

Query relevant existing entities before compiling a new source instead of relying on static top-50 lists.

Knowledge-hole detection

Use weak inter-community bridges to identify missing concepts that could connect topic clusters.

Learning paths

Start from a concept, traverse neighbors, then expand through communities.

Source audit

Ask which entities a source contributed and which relationships were created from it.

Concept provenance

Ask which raw sources currently support a concept.

Belief revision

Use the claim layer to track when new information reinforces, qualifies, contradicts, or supersedes prior beliefs.

Model evaluation

Evaluate whether a compile result integrates into the existing graph or creates disconnected islands.

Relationship search

Search over structure and provenance instead of only matching text.

Summary

GraphDB-KDB is the system's memory of relationships. It stores the ontology that the compiler builds from raw text, while Obsidian Markdown gives that ontology a readable surface. The value is not the graph visualization; the value is queryable structure, provenance, verification, and adaptive learning workflows on top of that structure.
