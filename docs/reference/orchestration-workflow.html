1The big picture

One conductor — kdb-orchestrate — runs the whole loop. The founding rule: the LLM is stateless compute and never touches the filesystem; Python owns every write, every path, every timestamp. The LLM only returns structured JSON.

The whole loop: per-source stages 1–9, then a single finalize pass per run.

The LLM call contract (both passes)

LayerAttemptsOn exhaustion
Transport — per API call (call_model_retry.py)3 — exponential backoff (1s → 30s cap, jittered, honors Retry-After); transient network / rate-limit / 5xx onlyerror surfaces to the content layer
Content — per source, per pass2 — deterministic repairs are free (no API call, don't consume attempts): stray-backslash escape; slug coercionfailure declared → quarantined: manifest error row (RUN_STATE_ERROR_INGEST for Pass-1, RUN_STATE_ERROR_COMPILE for Pass-2), event recorded, the loop continues — one bad source never kills the batch (D-96-1). Auto-retried next run: its hash never became last_compiled_hash.
Run level—only run-fatal / invariant failures abort (fail-fast, D-91-8); last_orchestrate.json is written success, abort, or crash

Who owns what

LayerOwns
LLMJudgment + prose. Structured JSON only: slugs, bodies, links. No paths, no timestamps, no frontmatter.
PythonAll state + I/O: scanning, hashing, validation, repair, page writes, graph transactions, ledger.
Kuzu graphThe live ontology authority: Entity, Source, Domain, LINKS_TO, SUPPORTS, ALIAS_OF, BELONGS_TO.
Markdown vaultPersistent state humans read: sources (with Pass-1 frontmatter) + rendered wiki pages.

Why this shape

  • No hallucinated paths — the model can't invent a filesystem location because it never emits one.
  • Every write is auditable + dry-run capable — all mutations flow through one Python commit sequence.
  • Crash-safe by construction — the manifest is written last, so any mid-source failure simply re-runs that source next time.
  • The graph is derivable — journals + compile_result.json rebuild it from scratch (graphdb-kdb rebuild, D39).
Run size is a design variable (vault-in-place note) finalize runs once per invocation, and its cost scales with the batch and the graph: batch wire_links over all accumulated pages, PageRank recompute, orphan sweep. For a full-vault ingest (~2k sources, 6–12h), prefer chunked runs (--limit N, repeated) over one monolith: each chunk finalizes over its own batch — bounding finalize cost and memory, producing per-chunk checkpoints and journals, and capping the #94 blast radius if a fail-fast fires mid-run. The open trade-off for the rollout design: per-chunk finalize overhead vs. monolith efficiency.

2Pipelines & scan scope

The system is multi-pipeline by design. One hand-authored registry per vault — <vault>/KDB/state/pipelines.json — declares which ingestion pipelines exist and what each one scans. kdb-orchestrate --pipeline <id> runs one pipeline per invocation.

Config, not code: pipeline behavior comes entirely from root + excludes + globs. The type field is validated but behaviorally inert — no special-casing anywhere.
Tracked design note (ratified direction for the rollout): deprecate the in-place vs raw concept entirely — there are only pipelines, each an absolute root plus scope globs. What we call "raw" pipelines are just roots at <vault>/KDB/raw/<pipeline-id> by convention. Companion guard: when pipelines.json is written/loaded, audit for overlapping roots — and the check must be exclude-aware (the vault root contains KDB/raw/* yet stays disjoint via excludes: ["KDB/"]; a naive prefix check would false-positive on exactly that).

Source identity

  • source_id = vault-root-relative path (kdb_scan.py _rel_to_vault) — never pipeline-relative, so no ambiguity and no cross-pipeline collisions.
  • The scan diffs against only this pipeline's manifest records (filtered by pipeline_id) — MOVED/DELETED never leak across pipelines (D-91-9).
  • Renames are detected by a two-pass hash match: same hash, new path → MOVED, not delete+add.

Configuration rules

  • Scopes must be disjoint. The manifest is keyed by source_id; two overlapping roots would double-ingest the same file under one key. (Enforcement = candidate validation work.)
  • excludes = walk-time prune — never hashed, never tracked. force_noise = still scanned and hashed, recorded as sources, but skipped by the LLM.
  • Feeder automation (RSS, podcasts, …) is the empty ingestion/feeder/ package — the registry's feeder slot is the reserved hook.

3Hashes & the manifest

SHA-256 is the heartbeat of incremental operation. One honest rule decides what compiles (D46): a file compiles iff current_hash ≠ last_compiled_hash. No force flags, no run_state games.

The post-embed hash override: the manifest always records the file as it sits on disk after Pass-1.

What this buys you in daily use

  • Edit-driven recompiles for free. Change a note in Obsidian → next run re-enriches and recompiles exactly that file. This is what makes vault-in-place living instead of a one-shot import.
  • Crash = resume. A killed run restarts cleanly: committed sources are hash-current and skipped; uncommitted ones recompile.
  • MOVED is cheap. Rename a file without editing it → hash match → path update, no LLM spend.

4Pass-1 — Enrich

One cheap LLM call per in-scope source. The model sees only the source body — never the file name or path — and returns one JSON envelope: triage, classification, and retrieval seeds. Python then validates, repairs, overrides, stamps, and embeds it as YAML frontmatter.

Pass-1: envelope → gauntlet → frontmatter. Two-stage validation (Task #95): raw LLM output first (retry-gated), assembled envelope after stamping (catches Python bugs).
Retries: 3 transport attempts per API call; 2 content attempts (rung-1 repair is free). Failure is declared after the 2nd content attempt → source quarantined (enrich_failed, RUN_STATE_ERROR_INGEST) and the loop continues.

### Boundary disambiguation rules

When content sits at a domain or source_type edge, apply these rules to
choose between candidates. (These are classification rules, not
relationships.)

**Domain boundaries:**

- ai-ml vs software vs hardware: compute stack — AI algorithms/models → ai-ml; OS/dev tools/programming → software; chips/silicon/electronics → hardware. Content classifies at the layer it primarily operates on.
- ai-ml vs software (AI tooling override): if content reads as software/dev-tooling BUT its subject is an AI/LLM model or product (Claude, ChatGPT, Codex, DeepSeek, Qwen, Gemini, Grok, Copilot, etc.) — classify as ai-ml, not software. Reserve software for general programming/OS/dev tooling with no AI/LLM subject.
- neuroscience-cognition vs biology: brain mechanisms, cognition (the former) vs cellular/genetic/evolutionary (the latter).
- psychology vs neuroscience-cognition: behavior/self-improvement/applied (the former) vs mechanism-level empirical brain science (the latter).
- health-wellbeing vs biology: applied personal-health decisions (the former) vs biological mechanisms without personal application (the latter).
- personal-finance vs value-investing: applied/situational (sector analysis, portfolio, tax) is personal-finance vs investment philosophy/methods/models is value-investing.
- value-investing vs economy-markets: investment-decision lens (what to buy/sell, valuation) vs market-mechanism lens (how markets/economies function).
- literature vs philosophy: narrative form (fiction, poetry) vs argumentative form (systematic thought).
- lifestyle vs personal-finance: experiential/personal-living (travel, hobbies, retirement activities) vs resource-management (retirement planning, tax, portfolio).
- lifestyle vs health-wellbeing: living-experiential focus vs health-focused (nutrition for longevity, fitness protocols).
- geopolitics vs history: current/recent → geopolitics; completed historical period → history.
- science-technology (catch-all): use ONLY when no specific S&T domain (#1-7) fits AND you can articulate why.

**Source_type boundaries:**

- blog vs post: blog = own publication (personal blog, Substack on own subdomain); post = community/forum/aggregator. When venue cannot be inferred, classify by authorial stance: self-contained piece → blog; community-participation → post.
- article vs news: article = analysis/argument/extended take; news = event reporting. Hybrid: classify by dominant mode.
- Transcript family: Q&A dominates → interview regardless of medium; one-direction educational delivery → transcript-lecture regardless of medium; otherwise medium-based (transcript-podcast vs transcript-video).
- book-chapter vs book-summary: chapter = verbatim book text; summary = ABOUT the book. Annotated excerpts classified by volume: verbatim majority → book-chapter; user-authored majority → book-summary.
- letter vs email: letter = curated public-facing addressed correspondence; email = informal individual/small-group.
- speech vs transcript-lecture: speech = prepared text form of address; transcript-lecture = transcribed-from-delivery.
- wiki vs article: wiki = encyclopedic register (third-person, neutral, multi-source citations); article = editorial (author voice, argument).
- social-thread vs post: social-thread = platform-native substantive authored content (multi-tweet thread, LinkedIn long-form post, substantive single-post platform-native essay); post = community comment or short casual share.
- interview vs meeting-notes: interview = verbatim Q&A (transcribed or text-native); meeting-notes = user-summarized.
- daily-note vs meeting-notes: daily-note = date-stamped omnibus log; meeting-notes = single-meeting dedicated artifact.
- documentation vs wiki: documentation = product/instructional reference (READMEs, runbooks, tutorials — reader does); wiki = descriptive encyclopedic entry (reader learns about).
- chat-log vs interview: chat-log = informal multi-party / human-to-AI exchange (no curated questioner/subject roles); interview = curated Q&A (clear interlocutor + subject).

### Content-description fields

- `author`: string or null. Extract the source's primary author from the
  content if attributable; null otherwise.
- `summary`: 1-3 sentences of plain prose distilling the substantive content.
  Neutral, descriptive register — no marketing or promotional phrasing.
- `key_themes`: list of 2-5 strings naming finer-grained sub-topics than
  `domain`. Format: lowercase kebab-case slugs (e.g. "value-investing",
  "attention-mechanism", "breathing-exercises") — not natural-language
  phrases with spaces.
- `entity_search_keys`: list of lowercase kebab-case slugs for the notable
  people, organizations, concepts, frameworks, and named ideas this source
  substantively engages. These seed a later entity lookup, so emit the
  canonical form a reader would search by. Guidance:
  1. Include each `key_themes` slug.
  2. Add slugs for entities named substantively in the source — people,
     organizations, named frameworks. Prefer the full-name form
     ("warren-buffett") over surname-only ("buffett"); emit only one form
     per entity.
  3. Add closely-related concepts the source actually engages as distinct
     ideas (e.g. a framework's foundational principle). Do NOT include
     speculative or weak co-occurrences.
  
  Emit only the most relevant keys — those that genuinely identify what this
  source is about. Prefer a few high-precision slugs over a long list; do not
  pad. Hard cap: 10. Format: lowercase, hyphens between words, no spaces, no
  punctuation other than hyphens (conjunctions/prepositions inside slugs are
  fine: "graham-and-doddsville", "theory-of-mind"). Prefer specificity:
  "value-investing" beats "investing".

  Examples (domain-diverse):

  - Finance: a source about Warren Buffett and Charlie Munger's value-investing
    approach (key_themes ["value-investing", "margin-of-safety", "compounding"])
    → `["value-investing", "margin-of-safety", "compounding",
       "warren-buffett", "charlie-munger", "intrinsic-value",
       "circle-of-competence"]`.

  - AI/ML: a source about transformer architectures in NLP
    (key_themes ["attention-mechanism", "self-attention", "transformers"])
    → `["attention-mechanism", "self-attention", "transformers",
       "multi-head-attention", "positional-encoding", "bert"]`.

  - Philosophy: a source about Rawls' veil of ignorance
    (key_themes ["veil-of-ignorance", "distributive-justice", "original-position"])
    → `["veil-of-ignorance", "distributive-justice", "original-position",
       "john-rawls", "theory-of-justice", "social-contract"]`.

### Confidence fields

- `confidence`: a number from 0.0 to 1.0 — your confidence in the `kdb_signal`
  call. Calibrate: 0.9-1.0 clear-cut; 0.7-0.89 reasonable with some ambiguity;
  0.5-0.69 genuine uncertainty between two options; below 0.5 essentially a guess.
- `uncertainty_reason`: string or null. Populate with the nature of the doubt
  whenever `confidence < 0.9` OR you resolved a close boundary call (e.g. a
  domain or source_type that was nearly a tie). Null otherwise.
- `reject_reason`: string or null. When `kdb_signal` is "noise", populate with
  the reason. Null otherwise.
- `other_reason`: string or null. Required (non-null) when `source_type` is
  "other" — name the specific missing publication form. Null otherwise.

# KDB Compiler — System Prompt

You are a semantic compiler for a personal Obsidian knowledge base. You read one source document and return one JSON object describing the wiki pages that source should produce. Ground everything in the source text — never invent facts, citations, URLs, dates, or author names. If the source does not support a claim, omit it.

---

## 1. Input

You will be given one source document from a markdown knowledge base. The full text of the source appears in the user message under `## SOURCE CONTENT`, preceded by `source_name:`. Treat that `source_name` as the canonical identifier for this source — echo it verbatim in your response. The source text itself is the only substrate for the claims you make about it.

Two more blocks appear in the user message:

- `## EXISTING CONTEXT (manifest snapshot)` — pages already in the knowledge base from sources compiled before this one, rendered without bodies (slug + title + page_type + outgoing_links). This is how you see what the knowledge base already contains, so that your response can connect to existing pages instead of creating duplicates. It may be empty.
- `## RESPONSE SCHEMA` — the JSON schema your output must conform to. If you are unsure about a field, re-read the schema rather than guessing.

You output **slug-space** data only — slugs, titles, bodies, wikilinks, and one `source_name` field that echoes the file you were given. **Source-id-space fields** (full path-prefixed `source_id`, per-page `supports_page_existence`, per-log_entry `related_source_ids`) are the runner's responsibility and are added to your output after the runner parses it. Do not invent path-prefixed source_ids; if the schema doesn't ask for one, don't emit one.

---

## 2. Example of what you return

The example below is the canonical shape. Every rule in the sections that follow refers back to it.

Imagine the user message carries a source named `attention-is-all-you-need.md` (the 2017 Transformer paper). Assume EXISTING CONTEXT contains two prior pages: `softmax` (concept, contributed by a general ML source) and `attention-mechanism` (concept, contributed by an earlier Bahdanau 2014 paper). Your response would be the JSON object below.

> **Annotations after `//` are teaching aids only — they MUST NOT appear in your output.** Your output is one JSON object with no comments.

```json
{
  "source_name": "attention-is-all-you-need.md",        // echo verbatim whatever you receive in the `source_name:` line; filename only, no path prefix
```

## 3. What pages to return

Identify the wiki pages this source should produce. A "wiki page" is one of three kinds:

- **`summary`** — one per source. A short overview of what this source is about and what concepts it introduces or reinforces. **The summary's slug is always `summary-<stem>` where `<stem>` is the source file's stem in kebab-case, copied verbatim — preserve every character of the stem, including numeric or identifier prefixes.** Examples: `KDB/raw/attention-is-all-you-need.md` → `summary-attention-is-all-you-need`; `KDB/raw/EP1 - The Journey of China.md` → `summary-ep1-the-journey-of-china`; `KDB/raw/04-research-debt.md` → `summary-04-research-debt`. The `summary-` prefix is reserved — it MUST appear on every summary slug and MUST NOT appear on concept or article slugs. Every compile returns exactly one summary page.
- **`concept`** — one per atomic idea. A concept is a reusable building block of the ontology; it may be supported by many sources over time. Prefer many small concept pages (one idea, one page) over few large ones. Use short, semantic, kebab-case slugs **that do not start with the reserved `summary-` prefix**. In the example: `attention-mechanism`, `self-attention`, `positional-encoding`.
- **`article`** — a narrative that ties several concepts together into a cohesive whole. An article is 1+1=3: its meaning is more than the sum of the concept pages it references — it captures something about how the source *uses those concepts together* that no single concept page, and no addition of concept pages, would carry on its own. In the example, `transformer-attention-only-architecture` is an article because the paper's claim is the combination — self-attention paired with positional encoding, as a standalone sequence model — and that claim is not on any of the individual concept pages.

### When to include an article

Include an article only when the source itself is doing this kind of synthesis — tying concepts together in a way that changes what they mean collectively, not just mentioning them in succession. Most sources don't: focused technical papers, interviews on a single topic, and how-to guides typically produce one summary and a handful of concepts, with no article. When you are on the fence, skip the article and let the concept pages stand on their own.

### Per-page metadata

For each page you include, commit to:

- **`confidence`** — an honest bucket:
  - **`high`** — the source states the material explicitly; no interpretation required. In the example, `self-attention` is `high` because the paper defines it directly.
  - **`medium`** — the source implies or assembles the material across passages; you synthesized rather than quoted. In the example, `positional-encoding` is `medium` because the paper gives the sinusoidal formula without a theoretical treatment — the page's framing ("self-attention is permutation-invariant, so without order signals a sentence would be a bag of words") is pieced together from surrounding context, not stated in the text.
  - **`low`** — the source gestures at the idea but doesn't develop it; you are reaching a little. When you land here, consider whether the page should exist at all, or whether a mention inside another page would do instead.
  - Three buckets are enough — there is no `0.72`. Pick the honest one. Do not stretch to `high` to seem decisive.
- **`status`** — always `active` for pages you are creating or updating in this compile.

## 4. Reuse slugs from EXISTING CONTEXT

Read each EXISTING CONTEXT entry by its *meaning*, not its spelling. For each existing page, ask: does this source discuss the same underlying idea? If yes, reuse the existing slug verbatim — in body wikilinks, in `outgoing_links`, and (if the page is being extended) in `concept_slugs` / `article_slugs`.

In the example, `attention-mechanism` is reused from EXISTING CONTEXT rather than minted as `attention-mech` or `self-attention-mechanism`. The Transformer paper's treatment of attention is the *same underlying concept* that Bahdanau 2014 introduced, even though the paper's phrasing and level of generality differ. That semantic sameness is what licenses reuse.

### When to reuse vs. mint a new slug

- **Reuse** when the existing page is about the same underlying concept, even if the source uses a different word, a different tense or number, or a different level of abstraction. If the ideas would live comfortably on the same page without one overwriting the other, they belong on the same slug.
- **Mint a new slug** when the source is discussing a genuinely distinct idea — a related cousin, a specific variant that deserves its own page, or a same-named concept from a different domain that shares the word but not the mechanism. When you mint a sibling to an existing slug, link to the existing slug explicitly in the new page's body so the relationship is legible. In the example, `self-attention` is minted as a sibling to `attention-mechanism` — close family, but a distinct enough idea to deserve its own page, and its body links back to `[[attention-mechanism]]`.
- **Genuinely on the fence** — when you cannot confidently tell whether the current source's idea is the same as or different from an existing slug, lean toward reuse. A missed reuse fragments the graph irreversibly; a shared slug that accumulates slightly broader meaning over time is self-correcting as more sources compile onto it.

Record non-obvious reuse or sibling decisions in `log_entries` (see the example's entry about extending `attention-mechanism`).

### Cold start (empty EXISTING CONTEXT)

Sometimes the block is empty — the source is the first in its domain, or nothing prior overlaps semantically. That is expected. When EXISTING CONTEXT is empty, you are building the ontology from scratch: mint slugs using the same conventions (short, semantic, kebab-case), and link concepts to each other inside this compile. Subsequent sources in the same domain will see what this compile returns and compound onto it.

## 5. Body format

Write page bodies in Obsidian-flavored markdown. Keep bodies focused, readable, and proportional to the page type:

- **Summary bodies** are short — a paragraph or two of overview, followed by `[[concept-slug]]` links to the concepts and articles this source contributed. Not a reproduction of the source; not a line-by-line rehash. A reader should skim the summary and know whether to open the source itself. The example summary models this.
- **Concept bodies** are atomic — explain the one idea as the current source presents it (treating any extension of an EXISTING CONTEXT page as adding to that page's accumulated treatment, not overwriting it). Link to related concepts via `[[slug]]`. If the source contradicts a prior source's treatment of the same concept, record that in `log_entries` rather than picking a winner in the body.
- **Article bodies** are narrative — weave the referenced concepts into a unified argument. Heavy linking is expected. Articles read like essays, not lists of bullets. The example article models this.

### Wikilink syntax

Use Obsidian-flavored wikilink syntax only:

- `[[slug]]` — plain link. (Example: `[[self-attention]]`.)
- `[[slug|display text]]` — link with custom display text, used when the sentence grammar wants a different surface word than the slug. (Example: `[[attention-mechanism|attention]]` — the slug is `attention-mechanism`, but the sentence reads "built entirely from attention".)
- `[[slug#heading]]` — link to a specific heading inside the target page. Use sparingly.

No HTML. No markdown-style `[text](url)` for internal slugs. No bare URLs for internal references.

### `outgoing_links` must match the body

For every slug in a page's `outgoing_links`, that slug must appear as `[[slug]]` somewhere in that page's body. And the reverse: every `[[slug]]` in the body must appear in `outgoing_links`. The two lists must agree exactly.

Every slug you use in `outgoing_links` must exist — either as a page you are returning in this response, or as an entry in EXISTING CONTEXT. Do not link to slugs that do not exist anywhere.

### Tone and length

- Prose, not bullet-lists, unless the source is inherently list-like.
- Present tense, neutral voice, third person.
- No meta-commentary about the compile process ("This page summarizes…", "As of this source…").
- No YAML frontmatter inside any `body` field. Body is pure content.
- Summaries: ~150–400 words. Concepts: ~100–500 words, scaling with the depth the source provides. Articles: ~400–1,200 words. Soft guides — shorter is fine when the source is thin; longer should be rare and justified.

## 6. Output envelope

Return one JSON object matching the schema in the user message. Nothing before it, nothing after it, no markdown fences around it, no prose explaining it.

The top-level object has these fields (see §2 for the canonical shape):

- `source_name` — echo the `source_name` you were given, verbatim. Filename only, no path prefix.
- `summary_slug` — the slug of the single summary page.
- `concept_slugs`, `article_slugs` — flat slug lists that mirror `pages[]`. Build each by walking `pages[]` and collecting every `concept` (resp. `article`) page's slug — every such page contributes exactly one entry. Equivalently, every entry in these lists must correspond to exactly one page of the matching type in `pages[]`. Three concept pages in `pages[]` → three entries in `concept_slugs` with the same slugs; no more, no fewer.
- `pages[]` — one entry per page you are returning, with the full set of fields shown in the example.
- `log_entries[]` — structured notes about this compile, each `{level, message, related_slugs}` (see §2). Use for notable decisions that deserve traceability: non-obvious slug reuse, choosing a sibling rather than reusing an existing slug, article-vs-concept boundary calls, contradictions between sources, concepts that almost-but-not-quite matched an existing slug. Empty array is fine. Levels: `info` for routine decisions, `notice` for something worth a second look, `contradiction` for conflicting claims between this source and a page's prior support, `warning` for anything the compile run should surface.
- `warnings[]` — free-text observations about the source itself (ambiguous terms, unresolved references, uncertain categorization). Empty array is fine.

### Thin or trivial sources

If the source genuinely contains nothing knowledge-worthy, return a single summary page whose body honestly says so, leave `concept_slugs` and `article_slugs` empty, and add a `warning` explaining why. Do not fabricate pages to fill a schema.

Malformed output aborts the run before any file is written.

Between the LLM's JSON and any durable write sit three deterministic stages. Their job: make whole defect classes structurally impossible before state is touched.

No LLM calls happen in stages 4–6 Every retry already happened inside the Pass-2 call (3 transport attempts; 2 content attempts wrapping parse/schema/semantic checks). Downstream of compile_one the model is done for that source — these stages only decide how to fail, never whether to retry: validate → quarantine (source-local) · repair → run-fatal invariant abort (code's own guarantee broke) · canonicalize → fatal halt before any write (D-R5-9; the fix is human — repair the alias ledger, no LLM sample can fix a circular alias).
StageModuleWhat it guarantees
4 · Validatevalidate_compile_result.py + per-source response schemaMalformed payloads never reach state. Per-source failure → quarantine; the batch survives (D-96-1).
5 · Repairrepair.pyreconcile_slug_lists rebuilds concept/article lists from pages[] (pages win); reconcile_body_links reconciles declared links against body wikilinks (body wins). Pairing inconsistency becomes unrepresentable.
6 · Canonicalizecanonicalize.py (Task #74)Alias surface forms resolve to canonical slugs via the human-maintained state/canonicalization/aliases.json ledger — wiki ≡ graph at the naming layer.

Canonicalize, up close

  • Ledger entries: surface → canonical (e.g. "AAPL" → "apple-inc"), chain-flattened to root.
  • Rewrites outgoing_links and body [[wikilinks]]; drops alias page intents; emits canonical_meta.aliases_emitted (later materialized as ALIAS_OF edges).
  • The ledger's SHA-256 is recorded into canonical_meta so a rebuild can prove it replayed against the same ledger state.
  • Algorithmic failures are fatal (circular aliases, malformed ledger, sha mismatch): the pipeline halts before page_writer (D-R5-9). Better no write than an inconsistent write.

7The commit sequence (β order)

All durable state changes for one source happen in a strict order, with the manifest written LAST — that write is the commit boundary. Anything that fails before it leaves the source looking un-compiled, so it simply re-runs next time.

β ordering (D-91-15): pages → graph → manifest. The order is chosen so every failure is recoverable without human intervention.

Terms used here (and across sections)

TermMeaning
bodyThe prose content of a wiki page — everything below the YAML frontmatter metadata block. The LLM writes bodies; Python writes frontmatter. (Field name in the LLM's JSON: pages[].body.)
whole-page rewrite (D18)A wiki page is never updated partially — every update replaces the entire page: the complete file, frontmatter and body together. (The decision ledger's name for this is "full-body replacement", because the LLM must always send the complete body text, never just a changed fragment — that's what makes rewriting the whole page possible. No diffing anywhere in the system.)
atomic write (D14)Never edit an existing file piecemeal. Instead (common/atomic_io.py): write the full new content to a temp file in the same directory → flush it to disk (fsync) → rename the temp over the target — one indivisible filesystem operation. Readers (Obsidian, Drive sync) always see a complete old file or a complete new one — never a half-written one.
txn / rolled backTxn = database transaction: the bundle of graph changes for one source (entities, edges, domains, aliases), executed all-or-nothing. BEGIN → make every change → COMMIT makes them real; if anything fails first, the database rolls back — undoes the whole bundle, leaving zero trace. The database analog of the atomic write.
idempotentDoing it twice has the same effect as doing it once. Writing the same page twice leaves the file exactly as if written once — no duplication, no corruption.
throw / raises"Fails with an exception" at that point in the sequence.
clean re-run (case-a)If a run dies before the manifest write, the bookkeeping still says "never compiled", so the next run simply compiles that source again. No leftovers to untangle: re-writing a page just replaces it (whole-page rewrite), re-adding graph nodes updates them in place, and this time the manifest lands.
commit boundaryThe manifest write — the last step of the commit sequence. Its content is prepared in memory beforehand (no I/O), so the final step is a single quick write. Before it: nothing officially happened, so re-running is free. After it: the source is officially compiled, its manifest row storing the post-embed hash H₁ as last_compiled_hash.
β ordering (D-91-15)"Option β" from the Task #91 design panel — not beta software. The blueprint's original pseudo-code was option α (manifest first, then graph); β (graph-sync first, manifest LAST) won unanimously because α allowed a "manifest says compiled but graph is empty" state after a sync failure. β eliminates that class by construction: graph failure → rollback → manifest never written → clean re-run.

page_writer guarantees

  • Pure render core + I/O shell: each page intent → PagePatch (slug → path, frontmatter, body).
  • Frontmatter carries provenance: title · slug · page_type · raw_path · raw_hash · compiled_at · source_refs.
  • Never writes outside KDB/wiki/ — sources and state are untouchable at this stage.

8Graph intake phases

Inside the per-source Kuzu transaction, six ordered phases mutate the graph. The ordering exists so references always resolve and projections derive from fresh state.

Two-pass entity ingest: all nodes exist before any edge is wired, so cross-entity (and cross-source) references always resolve. Any exception → ROLLBACK.
Why domains and aliases are phases, not afterthoughts BELONGS_TO is a derived projection of Source.domain × SUPPORTS (schema v2.4) — it must run after both exist in the txn. Alias Entities + ALIAS_OF edges come only from canonical_meta.aliases_emitted — the one per-source graph effect that lives outside compiled_sources[], which is why the finalize merge treats its union as load-bearing for replay fidelity.

9Finalize pass

Once per run, after the last source commits. This is where the deliberately-deferred work lands — and the ordering is what makes live ≡ replay hold by construction.

What finalize persists

  • state/compile_result.json — the combined replay payload (input to graphdb-kdb rebuild).
  • state/runs/<run_id>/ — per-run sidecars + journals (audit log; also the rebuild material, D39).
  • state/last_orchestrate.json — slim run summary: counts, failures, manifest delta (D-91-10).
  • Cleanup journal + retraction.json when orphans were reaped.

Six layers, each with one job (D51). The graph is the authority, but it is derivable — the journals are the real treasure.

11Failure semantics & operations

The system is designed so that failures are boring: each one has exactly one defined outcome, and almost all of them self-heal on the next run.

FailureOutcomeRecovery
Bad LLM output after 2 content attempts (parse/schema/semantic; free repair rungs in between; transport adds 3 backoff attempts per call)Source quarantined; batch continues (D-96-1)Recorded in journal + RespStatsRecord; re-runs next time automatically

Operator kill / crash mid-runCommitted sources are hash-currentNext run resumes from the first uncommitted source
Known pre-production blocker — Task #94 (fail-fast strands LINKS_TO) _finalize runs only when the run completes normally. A mid-run fail-fast abort (D-91-8) leaves already-committed sources with entities + SUPPORTS but zero LINKS_TO, and no compile_result.json. The next run sees those sources as hash-current → skips them → finalize wires only the new batch → the earlier sources' cross-source links are permanently lost.

Today's rule: on abort, delete graph + manifest and re-run clean — acceptable for a throwaway sandbox, unacceptable for a real vault. Fixing #94 (finalize-on-abort, per-source sidecars, or prefix-rewire) gates the first official run.

Operator surface

  • kdb-orchestrate --vault-root <vault> --pipeline <id> — run one pipeline. Omit --pipeline to list what's registered.
  • --dry-run — scan + print the plan; no LLM calls, no writes.
  • --limit N — stop after N compiled sources (noise is free); the remainder resumes next run. Finalize still runs over the compiled batch.
  • --model <pool-id> — model from common/models.json; --emit-kpis writes benchmark measurements for the run.
  • graphdb-kdb stats | verify | rebuild | snapshot — inspect, prove, reconstruct, or export the graph.
